Linker and Loader

I. Compiler, Linker, Loader

Compiler

source file → object file

Linker

여러 개의 object file을 결합해서 하나의 object file, 즉 executable file로 바꿔준다.

Loader

OS의 한 부분으로 executable file에 있는 내용을 메인 메모리에 올리고 프로세스로 만들어준다.

II. Linker의 역할

object file 안에는 여러 section이 존재한다.

Text Section (= Code Section)

실행할 instructions 가 위치한다.

ex) printf(”hello”);

Data Section

초기화 된 전역 변수들이 위치한다.

ex) int a = 0;

BSS Section

초기화 되지 않은 전역 변수들이 위치한다.

ex) int b;

ex) int array[1000000];

기타

이외에도 Symbol table, Relocation table 등의 섹션이 존재한다.

Note

왜 Data section과 BSS section을 구분하는 걸까?

초기화 된 전역변수는 변수의 주소 뿐만 아니라 초기값도 object 파일에 저장해야 한다. 예를 들어, int a = 0; 라는 instruction을 포함하는 파일은 최소 4 bytes 이상의 파일 용량을 가지게 된다.

하지만, 초기화 되지 않은 전역변수는 변수의 주소만 object 파일에 저장하면 된다. 예를 들어, int array[1000000]; 라는 instruction을 포함하는 파일이라고 해서 400MB 이상의 파일 용량을 가지지는 않는다. 이는 초기화 되지 않은 전역변수를 초기화 된 전역변수와 다르게 취급하기 때문이다.

Linker의 역할

링커는 여러 object 파일에 있는 동일한 타입의 섹션을 병합한다. 이 때, 어떤 주소에 어떤 섹션을 배치하는지는 링커 스크립트가 결정한다.

대부분의 개발자는 GNU 내장 링커 스크립트를 사용하기 때문에 이 부분에 대해서 고민할 필요가 없지만, embeded system 개발자의 경우 링커 스크립트로 섹션의 배치를 직접 세팅해야 될 때도 있다.

III. Loader의 역할

executable file에 어느 섹션을 어느 주소에 배치할지가 전부 정해져 있기 때문에 Loader는 executable file을 그대로 읽어서 메모리에 올려주면 그만이다.

이후 Program Counter만 프로그램의 시작점으로 세팅해주면 된다.

IV. Compiling and Linking Process

실제 compiling and linking process에서는 2가지 문제를 해결할 필요가 있다.

Problem 1

“Computer cannot recognize symbolic names”

컴퓨터는 사람이 지은 함수명이나 변수명을 이해할 수 없다.

이 문제는 컴파일링 단계에서 symbol name을 key, memory address를 value로 하는 symbol table을 만들면 해결된다.

Example of symbol table

int x = 10;
int y = 20;
int z = 30;
int m;

int func(int a) {
	return a+1;
}

int main(void) {
	m = x + y + z;
	x = func(10);
	return 0;
}
symbol namesectionoffset
functext0x0
maintext0x10
mbss0x0
xdata0x0
ydata0x4
zdata0x8

Problem 2

“Compiler can’t convert all symbol reference using symbol table”

컴파일러는 하나의 object file만 다룰 수 있기 때문에 cross-reference가 어느 주소에 있는지는 알 수가 없다.

이 문제를 해결하기 위해서 컴파일러는 memory address를 key, symbol name을 value로 하는 relocation table을 만들고 다음과 같은 일을 한다.

  • text section을 읽다가 internal symbol을 만나면 해당 symbol을 PC relative address로 변환한다.예를 들어, text section에 있는 call int_func instruction은 call PC+0x1B로 변환된다.
  • text section을 읽다가 cross-reference를 만나면 해당 symbol을 0x0으로 변환하고, relocation table에 (instruction’s address, symbolic name)을 등록한다.

    예를 들어, text section에 있는 call ext_func instruction은 call 0x0으로 변환된다.

Example of relocation table

extern int ext_var;
extern int ext_func(int a);

void main() {
	ext_var = 10;
	ext_func(10);
}
sectionoffsetsymbol name
text0x16ext_var
text0x24ext_func

링커는 링커 스크립트에 따라서 각 섹션을 병합하고 배치한다. 동시에 링커는 모든 object file의 symbol table을 병합한 symbol table을 만든다. 물론 final symbol table은 offset이 아닌 load address를 value로 갖는다.

마지막으로 링커는 relocation table과 symbol table을 이용해 cross-reference 0x0을 load address로 바꿔준다.

링커는 다음과 같은 일을 한다.

  • relocation table의 첫 번째 record를 읽는다.

    0x16에 있는 0x0의 symbol name은 ext_var이다.

    symbol table을 읽는다.

    ext_var의 load address는 0x8000이다.

    0x16에 있는 0x0을 0x8000으로 바꿔준다.

  • relocation table의 두 번째 record를 읽는다.

    0x24에 있는 0x0의 symbol name은 ext_var이다.

    symbol table을 읽는다.

    ext_var의 load address는 0x8000이다.

    0x24에 있는 0x0을 0x8000으로 바꿔준다.

  • relocation table의 세 번째 record를 읽는다.

    0x28에 있는 0x0의 symbol name은 ext_func이다.

    symbol table을 읽는다.

    ext_func의 load address는 0x4100이다.

    0x28에 있는 0x0을 0x4100으로 바꿔준다.

text section은 다음과 같이 변하게 된다.

V. Conclusion

Loader는 프로그램을 로딩하면서 4개의 segment를 생성한다.

  • Code Segment : Code Section
  • Data Segment : Data Section + BSS Section
  • Stack Segment
  • Heap Segment

code segment와 data segment는 프로그램을 로딩할 때 단 한 번만 메모리 할당을 한다.

즉, static memory allocation이 일어난다.

stack segment와 heap segment는 프로그램 실행 중에 필요할 때마다 메모리를 할당한다.

즉, dynamic storage allocation이 일어난다.
